## **Relevant Past Papers**

**NIKA — A Network Arena for Benchmarking AI Agents on Network Troubleshooting (arXiv:2512.16381)**

* *Summary:* A Kathará-based benchmark (640 incidents, 54 issue types, five scenarios) that evaluates LLM agents on network troubleshooting using a static pipeline (start → inject → agent → evaluate) and an MCP Agent Access Layer of 30+ tools. RCA is scored automatically: the agent submits root-cause name(s) chosen from a known list, and the evaluator does an exact match against ground truth.  
* *Gap:* Explicitly names tool-interface design (granularity and abstraction of external APIs) as an open dimension of the agent design space but never studies it. Their own error analysis observes agents stopping at "shallow, connectivity-centric explanations" instead of tracing symptoms to deeper causes — the exact failure mode we isolate.

**Ericsson / Aalto Bachelor's Thesis (Dinh Bach Tue) — MAS with MCP and A2A for transport network automation**

* *Summary:* Builds a multi-agent system using MCP and A2A protocols to automate transport network Q\&A tasks.  
* *Gap:* Flags tool overload and a 15.52s per-answer latency as problems but never quantifies how tool exposure affects accuracy; evaluated on only 15 scenario tests (3 trials each) with a single LLM vendor (OpenAI GPT-4o / GPT-4o-mini). ✅ Verified against the thesis (Table 2, Sec. 5–6).

**An LLM-based Agentic Framework for Accessible Network Control (arXiv:2509.20600)**

* *Summary:* An agentic framework for controlling networks through natural language.  
* *Gap:* Identifies retrieval as the dominant accuracy lever but does not isolate tool granularity or cross-layer (overlay→underlay) attribution as a measured variable.

## **Motivation**

* **The operational setting:** SD-WAN — the dominant enterprise WAN architecture — is by construction an overlay/underlay system: encrypted tunnels (the overlay) stitched across transport networks (the underlay). The classic SD-WAN operations question is "is it the overlay or the underlay problem?" — the symptom appears at the tunnel or application level, but the cause frequently sits in transport routing underneath.  
* **The problem:** When a fault shows up as an overlay symptom (e.g., a host cannot reach a service over a VPN tunnel), an LLM troubleshooting agent may stop at the symptom rather than trace it to the true underlay root cause. We do not know whether *how MCP tools are packaged* (coarse vs. fine granularity) changes the agent's ability to make that cross-layer attribution. As vendors expose network operations to AI agents through MCP, tool packaging is a design choice every one of them is making right now — without measurement.  
* **How we know it exists:** NIKA lists tool-interface design as an unstudied open dimension, and its authors directly observed agents settling on vague, surface-level diagnoses for incidents with indirect symptoms. The Ericsson thesis names tool overload as a problem in a transport-network automation setting but never measures it. Neither isolates cross-layer attribution.  
* **Why it matters:** Real SD-WAN and transport-network failures are routinely cross-layer, and tool granularity is a design knob operators and vendors actually control. If granularity measurably moves attribution accuracy, that is an actionable finding for how networking MCP tools should be exposed to agents.  
* **Why our idea solves it:** Paired *confusable failures* (identical overlay symptom, different underlay cause) make cross-layer attribution the only path to a correct answer, isolating it as a single measurable variable. Holding everything fixed except tool packaging isolates granularity.  
* **Why it would probably work:** NIKA's static pipeline supports this postmortem design with no harness changes ✅ (fault is injected once at session setup, before the agent connects; confirmed in `src/nika/workflows/failure_inject.py`). The `rip_small_internet_vpn` scenario reproduces the essential SD-WAN structure in its minimal form — an encrypted WireGuard tunnel (overlay) over a RIP-routed transport mesh (underlay) — and one half of our confusable pair already exists in NIKA as the registered fault `host_vpn_membership_missing` ✅.

## **Key Ideas / Contributions / Novelty**

* **Novelty:** First controlled measurement of how MCP tool granularity affects an LLM agent's cross-layer root-cause attribution in an overlay/underlay network. NIKA and the thesis both gesture at tool design / overload; neither quantifies it, and nor targets cross-layer attribution.  
* **Main contribution:** Not a new method but a measurement. The result is the **interaction**  does coarse vs. fine tool packaging change the rate at which the agent correctly traces an overlay symptom to its underlay cause, given identical information in both conditions?  
* **Operationalization (the construct):** We replace "is the agent smart enough" with cross-layer attribution accuracy, scored 1/0 per case by NIKA's built-in exact-match RCA evaluator (details in Evaluation). Correct root-cause name \= pass; wrong name or hedged double-pick \= fail.

## **Methods**

Postmortem-on-settled-state (NIKA's standard flow). For each incident the fault is fully in place before the agent connects; the agent diagnoses using only the tool set for its assigned condition; the agent submits its answer through NIKA's `submit()` tool, choosing root-cause name(s) from the list returned by `list_avail_problems()`; NIKA's RCA evaluator scores the submission by exact match against ground truth ✅.

**Testbed as a minimal SD-WAN abstraction:** the `rip_small_internet_vpn` scenario provides one encrypted tunnel over one routed transport — the smallest unit that preserves the overlay/underlay split every SD-WAN is built on. Confusable pairs exploit exactly that split: tunnel-config causes vs. transport-routing causes producing the same overlay symptom.

Two tool conditions, **information-matched** (every fact reachable in one is reachable in the other; only packaging differs):

* **FINE:** atomic named tools \- `get_tunnel_status` (overlay), `frr_show_ip_route` / `frr_show_running_config` / `frr_exec` — restricted to the show-command whitelist matching check\_underlay’s bundle (underlay), `get_reachability` / `ping_pair` / `curl_web_test` / `get_host_net_config` (symptom), `systemctl_ops` (action).  
* **COARSE:** per-layer aggregates \- `check_overlay`, `check_underlay`, `check_endpoint` \- each bundling the corresponding atomic calls. check\_underlay bundles frr\_show\_ip\_route, frr\_show\_running\_config, and RIP neighbor/interface status for each router — the same set frr\_exec is restricted to in FINE (see below). Crossing the layer boundary stays a deliberate call. This mirrors how SD-WAN management platforms typically expose health: per-layer rollups (overlay health, underlay health) rather than raw per-protocol commands.

Three controls baked into the design:

1. `exec_shell` / `exec_shell_dual` / `cat_file` (raw escape hatches) **removed from both conditions** — otherwise the agent bypasses the aggregates and the manipulation collapses.  
2. `get_tunnel_status` is a new named tool used in **both** conditions, added because the overlay otherwise has no native tool — this keeps the two layers tooling-symmetric so granularity is the only axis moving.

3. `frr_exec` in FINE is **restricted to a fixed whitelist** of show commands (ip route, running-config, RIP neighbor status, interface state) — exactly the commands `check_underlay` bundles in COARSE; it cannot run arbitrary router commands. This makes the information-matched claim for the underlay layer verifiable by comparing two fixed lists — a reviewer can check it by reading the lists rather than trusting our judgment. (The fairness check in Phase 3 still empirically diffs the overlay and endpoint layers.)

**Framing disclosure (stated openly):** Because the agent selects from a known list of candidate root causes, our task measures *forced-choice discrimination* between confusable causes, not open-ended diagnosis from scratch. This is the skill we intend to measure — telling two look-alike causes apart requires crossing the layer boundary — but we state it explicitly.

Implementation: a `kathara_coarse_mcp_server.py`, selected in `mcp_servers.py` via an env var (`TOOL_GRANULARITY=coarse|fine`). One new problem class per underlay-rooted fault (RIP routing faults producing the same overlay symptom as the tunnel-side fault); NIKA's problem pool auto-registers new problem classes ✅. No agent or pipeline changes. The LLM judge is used only for secondary trace analysis; if we adapt its rubric, that is a one-string template edit in `src/agent/utils/template.py` and does not affect the primary metric. (Note: the judge code has a latent bug — it overwrites its own prompt template after the first call — which we must patch if we batch-evaluate. ✅)

## **Experimental Setup**

* **Design:** Within-design comparison — FINE vs COARSE run on the **same** confusable-failure pairs with the **same** model. The FINE condition is the reference granularity (control).  
* **Model:** GPT-5-mini (locked). One model end-to-end first; a second model (e.g., GPT-5) only if time and budget allow. GPT-5-mini matches NIKA's published setup, enabling sanity checks against their numbers.  
* **Scenario:** `rip_small_internet_vpn` (single WireGuard tunnel over a RIP full-mesh transport underlay) as a minimal SD-WAN abstraction. No new multi-path SD-WAN topology in v1.  
* **Scale:** 10 confusable pairs × 2 cases × 2 conditions × 3 repeats \= **120 agent runs**. For reference, NIKA's full evaluation (150 incidents, GPT-5-mini) took \~10 hours, so this is roughly a day of machine time.  
* **Primary metric:** Cross-layer attribution accuracy (1/0 per case), from NIKA's built-in exact-match RCA evaluator.  
* **Confound control / ablation:**  
  * Information-matched conditions, verified by diffing the facts reachable in each before any run.  
  * Raw escape hatches removed from both conditions.  
  * `get_tunnel_status` symmetric across conditions.  
  * 3 repeat runs per case to bound run-to-run agent variance (LLMs are stochastic); NIKA's precedent was 2 runs per incident.  
* **Additional analysis:** Break accuracy out by failure type (underlay-rooted vs overlay-rooted) to surface the interaction; report the per-condition gap with run-to-run variance. LLM-judge trace ratings (relevance, correctness, efficiency, clarity) as supporting evidence on *how* reasoning differs between conditions.

## **Datasets and Evaluation**

* **Dataset:** NIKA's `rip_small_internet_vpn` scenario plus 10 paired confusable failures built on NIKA's problem-class mechanism. Each pair \= one underlay-rooted case \+ one overlay-rooted case sharing an identical overlay symptom. The overlay-rooted fault type exists ✅; each underlay-rooted twin is a new problem class we write. (Pair count adjustable if fault-design or runtime budget demands.)

### **Candidate confusable fault pairs (design table, pre-implementation)**

Design rule: viewed only from the tunnel layer, both faults in a pair must look identical; the difference must only be visible by inspecting the layer below. ✅ \= fault mechanism already exists in NIKA in some form.

| \# | Shared symptom (tunnel view) | Tunnel-side cause (overlay) | Routing-side cause (underlay) | Status |
| ----- | ----- | ----- | ----- | ----- |
| 1 | Tunnel dead, no handshake | Peer deleted from server's wg config ✅ | Edge router stops advertising server's subnet in RIP | Strong — verify mesh cannot reroute around the withdrawal |
| 2 | Pings pass, large transfers hang | Wrong MTU on wg0 | Transit link MTU lowered \+ ICMP "frag needed" blocked | Strong |
| 3 | Handshake sent, never answered | Wrong peer public key | Firewall drops UDP 51820 on a transit router | Strong |
| 4 | Traffic flows one way only | allowed-ips wrong on one peer | Blackhole route on the return path | Strong |
| 5 | Tunnel up and down repeatedly | WireGuard service crash-looping | Link flap on transit ✅ | Flagged — flapping is ongoing behavior; confirm it fits the settled-state design |
| 6 | No handshake; endpoint silent | Wrong endpoint IP in client's wg config | Server's public interface IP misconfigured ✅ | Strong |
| 7 | Tunnel dead, no handshake | WireGuard service stopped on server | Client host has blackhole route to server's public IP ✅ | Strong |
| 8 | Handshake never answered | Server listen port changed, client config stale | Firewall blocks that specific port | Flagged — near-twin of pair 3 |
| 9 | Tunnel up but slow / lossy | VPN server CPU exhausted (stress-ng) ✅ | Packet loss on transit link via tc ✅ | Decent — justify "CPU \= overlay-side" in one sentence |
| 10 | Erratic, on-off reachability | Two peers share the same inner tunnel IP | Server's public IP conflicts with another host ✅ | Flagged — symptom identity uncertain |

**Build order:** pairs 1–7 \+ 9 first (8 credible pairs); build 8 and 10 only if the first eight pass the symptom-identity hand-test (Phase 2). Eight verified pairs beat ten shaky ones. Every pair must pass the hand-test — inject both halves, run identical tunnel-level checks, confirm indistinguishable outputs — before entering the run set.

* **Training:** None — inference-only.  
* **Evaluation metric:** Cross-layer attribution accuracy (1/0), scored automatically by NIKA's exact-match RCA evaluator ✅. Scoring rules (closed, no edge cases left):  
  1. Submits only the correct root-cause name → **pass**.  
  2. Submits a wrong root-cause name → **fail**.  
  3. Submits both candidate causes (hedging) → **fail** — hedging means the agent could not discriminate, which is the construct under test.  
* **Secondary evaluation:** NIKA's LLM judge (default GPT-5-mini) rates the reasoning trace on relevance, correctness, efficiency, and clarity (1–5 each); used descriptively, never for the headline number.

## **Ideal Results**

* **Hypothesis:** Tool granularity changes cross-layer attribution accuracy even though both conditions expose identical information — i.e., packaging, not access, moves the agent's ability to cross the layer boundary.  
* **Best case:** A clear, non-obvious interaction — e.g., granularity barely affects single-layer attribution but swings cross-layer attribution by a meaningful margin. That interaction, not either main effect, is the contribution — and it translates directly into guidance for how SD-WAN and network-management vendors should package MCP tools for AI agents (per-layer rollups vs. atomic commands).  
* **A null result is still publishable:** "granularity does not affect attribution when information is held constant" is itself a finding, given the literature assumes tool design matters.

## **Execution Plan (7 Phases)**

**Phase 0 — Lab bring-up.** Clone NIKA, install Kathará \+ Docker, set the OpenAI API key. Run one *existing* incident in `rip_small_internet_vpn` with stock tools, end to end (agent connects → diagnoses → submits → gets scored). Goal: prove the pipeline works before changing anything. Budget a weekend for environment friction.

**Phase 1 — Build the tool conditions.** Three coding tasks: (1) new `get_tunnel_status` tool, added to both conditions; (2) `kathara_coarse_mcp_server.py` with the three per-layer aggregates (`check_overlay`, `check_underlay`, `check_endpoint`); (3) the `TOOL_GRANULARITY=coarse|fine` switch in `mcp_servers.py`, plus removal of `exec_shell` / `exec_shell_dual` / `cat_file` from both conditions. Test by listing the tool menu the agent sees in each mode — no diagnosis yet.

Task (NEW): Make the tool condition recoverable from results.

TOOL\_GRANULARITY selects which MCP servers load, but is not recorded in NIKA's output — so FINE and COARSE runs are indistinguishable in evaluation\_summary.csv as shipped. Add tool\_granularity as a recorded field: (1) session.py init\_session reads the env var into run.json; (2) add tool\_granularity to the EvalResult dataclass in result\_log.py; (3) populate it in build\_eval\_result\_from\_session\_dir. Result: a tool\_granularity column (fine/coarse) in the summary CSV to group the primary metric by

**Phase 2 — Build the faults (longest phase, \~2–4 weeks).** Implement each routing-side fault from the pairs table as a new NIKA problem class. Hand-test every pair: inject, confirm from a host that the symptom is indistinguishable from its pair partner's, confirm recovery restores the network. Build order: pairs 1–7 \+ 9 first; 8 and 10 only if the first eight pass. A bad pair poisons the results — do not rush this phase.

**Phase 3 — Fairness check (information matching).** Per pair: inject, run every FINE tool by hand and save outputs; run every COARSE tool and save outputs; diff. Every fact visible in one condition must be visible in the other; fix the coarse tools where they hide something. Keep the comparison files as evidence — this is the defense against the "coarse tools just had less information" objection. For the underlay layer this reduces to a structural check (the frr\_exec whitelist must equal check\_underlay’s bundled commands); the empirical diff is what’s left to do for the overlay and endpoint layers.

Phase 3 reminder — resolve the FINE-vs-COARSE tool-surface gap (carried over from Phase 1 wiring). As wired, FINE \= base \+ frr servers, which exposes diagnostic tools the three coarse aggregates do not cover: systemctl\_ops, get\_tc\_statistics, netstat, ip\_addr\_statistics, ethtool, iperf\_test, and the BGP/OSPF config tools (frr\_get\_bgp\_conf, frr\_get\_ospf\_conf). So FINE currently has a larger information surface than COARSE — the exact "coarse just had less info" objection this phase defends against. Decide per tool: trim from FINE, absorb into a coarse aggregate, or justify in writeup as not aiding cross-layer attribution. (systemctl\_ops is the most defensible to keep asymmetric, since check\_overlay's wg show already gives COARSE the equivalent WireGuard service-state signal.) Resolve via the hand-diff, not by guessing.

Also confirm here: the check\_endpoint web-test policy (currently the superset: each client curls each web server over the tunnel) — keep or narrow once the diff shows whether the superset is necessary.

**Phase 4 — Pilot.** One pair, both conditions, 2 repeats each \= 8 runs, watched closely. Checks the plumbing: submission works, exact-match scorer returns 1/0, traces persist, judge runs (after patching the prompt-template bug). Also the test-difficulty check: if both conditions ace the pilot, the pairs are not confusable enough — harden them *before* the real runs. Expect to find something broken; that is the pilot's job.

**Phase 5 — Full runs.** 10 pairs × 2 cases × 2 conditions × 3 repeats \= 120 runs (\~1 day machine time at NIKA's reported GPT-5-mini pace). Scripted and unattended; every trace and score saved. No mid-run changes — if a bug surfaces at run 60, fix it and restart from zero so all runs stay comparable.

**Phase 6 — Analysis.** Headline number: attribution accuracy per condition. Key breakdown: underlay-rooted vs overlay-rooted cases — the predicted interaction lives there, and a modest overall gap can hide a large gap in one slice. Report run-to-run variance; use the LLM judge's 1–5 trace ratings, tool-call counts, steps, and tokens as supporting evidence on *how* reasoning differed (also the fallback story if accuracy shows no gap). These metrics (tool\_calls, steps, tokens, time\_taken) and the LLM judge scores are produced automatically per session via NIKA's existing AgentCallbackLogger/AgentTraceParser pipeline — no additional instrumentation needed ✅ (confirmed in src/agent/utils/loggers.py and src/nika/evaluator/trace\_parser.py). 

Extra Note \- NIKA also computes detection\_score and localization\_accuracy automatically for every run, alongside RCA. can be added optionally.

**Phase 7 — Write-up.** This document is the methodology skeleton; add results, keep the limitations as written, hand to mentor and collaborator.

**Schedule risk note:** the entire schedule risk is Phase 2; everything else is days. If time gets tight, cut pairs (10 → 8 → 6\) — never cut repeats or the fairness check. Phases run strictly in order: any agent run made before Phase 3 completes is a run that gets thrown away.

## **Potential Limitations**

* **Not a full SD-WAN:** The testbed is the minimal overlay/underlay unit — one tunnel, one transport. It has no second transport path, no SD-WAN controller, and no application-aware path steering. Results inform SD-WAN tool design but do not benchmark a complete SD-WAN deployment; multi-path topologies are future work.  
* **Generalization:** Single scenario (`rip_small_internet_vpn`) and single model (GPT-5-mini) in v1 — limits external validity.  
* **Statistical power:** 10 pairs is modest; the per-condition gap may be noisy. Repeat runs mitigate but do not eliminate this.  
* **Forced-choice framing:** The agent picks from a known list of candidate causes, so the task measures discrimination between confusable causes, not open-ended diagnosis. We disclose this; suppressing the list would require modifying NIKA's task interface and is out of scope for v1.  
* **Static design:** Postmortem-on-settled-state covers no temporal/failover behavior.  
* **Coarse-tool design bias:** Aggregate boundaries could subtly favor one layer; the information-diff check is the guard, but the carve-up remains a judgment call.  
* **Ethical:** Low — simulated lab environment, no real network or user data. **\[VERIFY: venue requirements once target venue is chosen\]**
